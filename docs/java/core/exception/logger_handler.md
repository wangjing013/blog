# 理解 Logger 和 Handler 

如果把日志比作**自来水**，那么 **Logger** 就是**总水阀**，而 **Handler** 就是各个房间的**水龙头**。

我们可以把整个日志输出流程拆解为“生产、过滤、分发”三个步骤。

## 1. Logger：日志的“准入大门” (总水阀)

**Logger** 是你在代码里直接调用的对象（比如 `logger.info()`）。它的核心职责是**判断这条日志是否有资格产生**。

*   **角色**：日志的记录器。
*   **过滤逻辑**：它会检查你当前调用的级别（如 `FINE`）是否达到了它设置的门槛（`.level`）。
*   **层级关系**：Logger 是有父子关系的。如果你没给某个 Logger 设置级别，它会去问它的“爸爸”（父级 Logger），直到找到根日志（Root Logger）。

> **关键点**：如果日志级别低于 Logger 的门槛，日志请求在这里就会被直接丢弃，后续的 Handler 连一个字节的信息都收不到。

## 2. Handler：日志的“出口大门” (水龙头)

**Handler** 负责决定日志最终去哪儿（控制台、文件、网络等）。一个 Logger 可以挂载多个 Handler，就像一根水管可以接多个水龙头。

*   **角色**：日志的处理器。
*   **过滤逻辑**：即使 Logger 放行了日志，每个 Handler 还有自己的“二次过滤”权力（`handler.setLevel()`）。
*   **职责分工**：
    *   `ConsoleHandler`：把日志印到屏幕上。
    *   `FileHandler`：把日志写进硬盘文件。

> **关键点**：Handler 的级别不能比 Logger 更低。如果 Logger 只放行 `INFO`，你把 Handler 设为 `FINE` 也是没用的，因为 Logger 那关就没过。

## 3. 完整的“两道门”过关流程

我们可以通过下面这个逻辑链条来看看一条 `FINE` 日志的“闯关”之旅：

1.  **发起请求**：你在代码中执行 `logger.fine("Hello")`。
2.  **第一道门 (Logger)**：
    *   检查配置：`logger.getLevel()` 是多少？
    *   如果是 `FINE` 或更低 $\rightarrow$ **通过**。
    *   如果是 `INFO` $\rightarrow$ **拦截**（流程结束）。
3.  **传递阶段**：Logger 将日志对象包装好，分发给它身上挂载的所有 **Handlers**。
4.  **第二道门 (Handler)**：
    *   以 `ConsoleHandler` 为例，检查自己的级别。
    *   如果它的级别也是 `FINE` $\rightarrow$ **通过**。
    *   如果是 `INFO` $\rightarrow$ **拦截**（屏幕不显示，但如果同时挂了文件 Handler 且级别够，文件里会有）。
5.  **最终输出**：Handler 调用 **Formatter**（格式化器）把日志变成一串漂亮的文字，吐给控制台。


### 总结对比

| 维度 | Logger (日志记录器) | Handler (日志处理器) |
| :--- | :--- | :--- |
| **数量** | 通常每个类对应一个 | 一个 Logger 可拥有多个 |
| **职责** | **审判**（这行日志该不该发？） | **执行**（这行日志发到哪去？） |
| **层级** | 具有父子继承关系 | 附着在 Logger 身上 |
| **打个比方** | 报社的**总编**（决定哪些稿件通过） | 不同的**发行渠道**（报纸、网站、广播） |
