# 深度解析 SOCKS5 协议原理

SOCKS5 是一种位于``会话层（Session Layer）`` 的代理协议，它介于``传输层（TCP/UDP）``与``应用层（HTTP/FTP）``之间。相比 HTTP 代理，它更通用，能转发任何 TCP/UDP 流量。

SOCKS5 通信协议全生命周期：

* 握手阶段 (Method Negotiation)
* 身份认证阶段 (Authentication)
* 请求阶段 (Request Phase)
* 转发阶段 (Forwarding / Relay)

## 一、 握手阶段（Method Selection）

客户端连接服务器后，发送第一个二进制包，协商认证方式。

### 1. 客户端请求

**格式：** `[VER, NMETHODS, METHODS]`
*   **NMETHODS:** 表示几种认证方式，也就是 ``METHODS`` 的长度。
*   **METHODS：** 
    *   `0x00`: 不需要认证
    *   `0x01`: GSSAPI
    *   `0x02`: 账密认证
    *   ``0x03 - 0x7F``: 由IANA分配（保留）
    *   ``0x80 - 0xFE``: 为私人方法保留
    *   ``0xFF`` 无可接受的方法
*   **示例：** `[0x05, 0x01, 0x02]`
*   **含义：** 版本 5，支持 1 种方法，该方法为 `0x02`（用户名密码认证）。

### 2. 服务端响应

**格式：** `[VER, METHOD]`
*   **示例：** `[0x05, 0x02]`
*   **含义：** 接受请求，并选定使用用户名密码认证。

## 二、 身份验证阶段（Authentication）

一旦选定了 `0x02` 方法，后续逻辑进入 **RFC 1929** 定义的子协议。

### 1. 客户端提交身份
**格式：** `[子协议版本, 用户名长度, 用户名, 密码长度, 密码]`
*   **示例：** `<Buffer 01 05 61 64 6d 69 6e 06 31 32 33 34 35 36>`
*   **解析：** 子协议 v1，用户名 admin (5字节)，密码 123456 (6字节)。

通过一个代码，来解析出示例中的内容：

```js
const buff = Buffer.from([0x01, 0x05, 0x61, 0x64,0x6d,0x69,0x6e, 0x06,0x31 ,0x32 ,0x33, 0x34 , 0x35,  0x36]);
// Buffer 可以通过 bytes 数组来初始化，每个 byte 的值为 0 ～ 255 之间。
// 等价于八位二进制: 0000 0000 - 1111 1111 
// 等价于两位16进制: 0x00 - OxFF 
// 上面 Buffer 中传入内容就是两位的16进制的数组

const version = buff[0]; // 版本号
const usernameLen = buff[1]; // 5
const usernameLastPos = 2 + usernameLen;
const username = Uint8Array.prototype.slice.call(buff, 2, 2 + usernameLastPos).toString(); // admin

// 思考为什么解析出来就是 admin ?
// 只要理解 0x61 0x64 0x6d 0x69 0x6e 代码是什么？ 
// 第一步：通过 Uint8Array.prototype.slice.call 得到 Unit8Array 为 [0x61, 0x64, 0x6d, 0x69, 0x6e]
// 第二步：toString ，调用 toString 为什么是会变成 admin ？
// 通过``toString`` 方法，将 ``Unit8Array`` 中的内容通过 ``UTF-8`` 编码方式去转为字符串。

// 计算机底层质层只有 0、1 二进制数字。如果将 ``1010`` 展示给用户看时，用户肯定读不懂。

// 例如：有没有一种方式，通过将 ``1010`` 表示值，对应到一个特定数字、文字、符号等等，这样用户就能读懂。
// UTF-8(编码规则) + Unicode(字符集)
// Unicode 是一个映射表。它给世界上几乎所有的字符（字母、汉字、Emoji 等）都分配了一个唯一的数字，
// 这个数字被称为 码点 (Code Point)。
// Unicode它的作用： 规定了 a 是 97（十六进制 U+0061），“中” 是 20013（十六进制 U+4E2D）。它不负责该数字在计算机内存里到底是用 1个字节、2个字节还是4个字节来存。

// 具体几个字节是由 UTF-8 编码来确定的。
// UTF-8 采用 1 - 4 个字节来存储。
// 01 认为占用一个字节，直接将其转换为十进制值去对应的 Unicode 表中查找
// 110xxxxx 表示多字节首部 11000000 - 11011111 0xC0 ~ 0xDF
// 1110xxxx 表示多字节首部 11100000 - 11101111 0xE0 ~ 0xEF
// 11110xxx 表示多字节首部 11110000 - 11110111 0xF0 ~ 0xF7
// 10xxxxxx 不能作为首部，只能作为多字节的后续部分，也就是他不能独立存在。

// 再回 Buffer([0x61, 0x64, 0x6d, 0x69, 0x6e]).toString() 背后是隐藏几件事？
// toString 默认采用 UTF-8 编码，那么在转为完整字符串之前。
// 1、先判断 Buffer 里面元素，如果 01 开头也就是单字符，则直接转为码点(十进制)，然后去 Unicode 表中查找
// 2、如果发现多字节，那么就会继续去找下一个元素，直到下一个首部元素出现时终止。然后转换为码点去 Unicode 表中查找
// 另外，通常出现乱码原因就是，本来是三字节的字，我们把它截断，然后去 toString 就会出现乱码。
```

### 2. 服务端返回结果

**格式：** `[子协议版本, 鉴定状态]`
*   **认证成功：** `[0x01, 0x00]`
*   **认证失败：** `[0x01, 0x01]`（或其他非 0 状态）

## 三、 请求阶段（Requests）—— 关键一步

身份验证通过后，客户端会发送 **“代理目标指令”**。服务器需要解析这个包，去连接真正的目标服务器。

### 1. 客户端请求连接
**格式：** `[VER, CMD, RSV, ATYP, DST.ADDR, DST.PORT]`
*   **VER**：固定 `0x05`。
*   **CMD**：命令类型。
    *   `0x01`: CONNECT (最常用，建立 TCP 连接)。
    *   `0x02`: BIND。
    *   `0x03`: UDP ASSOCIATE。
*   **RSV**：保留字段，固定 `0x00`。
*   **ATYP**：目标地址类型。
    *   `0x01`: IPv4 (后面跟 4 字节 IP)。
    *   `0x03`: 域名 (后面第一个字节是长度，接着是域名字符串)。
    *   `0x04`: IPv6 (后面跟 16 字节 IP)。
*   **DST.ADDR & DST.PORT**：目标地址和 2 字节端口（大端序）。


通过一个案例理解下

```js
const buff = new Buffer([0x05 0x01 0x00 0x01 0x1c 0x00 0x00 0x11 0x00 0x50]);
// 按上面描述来解析这部分内容：
// 0x05 是协议版本
// 0x01 表示通过 CONNECT 方式连接
// 0x00 固定部分
// 0x01 表示为 IPv4
//      0x1c 28
//      0x00 0
//      0x00 0
//      0x11 17
// IP为：28.0.0.17
// 0x00 0x50 合并到一起，0x0050 对应值为 80
// 解析地址：28.0.0.17:80
```

### 2. 服务端响应

服务器尝试连接目标后，回包告诉客户端结果：

**格式：** `[0x05, REP, 0x00, ATYP, BND.ADDR, BND.PORT]`
*   **REP**：响应状态（`0x00` 代表连接成功，``0x01`` 代表连接失败）。

```js
// 连接成功的响应
[0x05, 0x00, 0x00, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
// 连接失败的响应
[0x05, 0x01, 0x00, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
```

## 四、 转发阶段（Relay）

至此，``SOCKS5`` 握手完全结束。

**逻辑转变：**

此时，服务器不再解析任何 `0x05` 或 `01` 开头的内容。它进入“盲转发”模式：

1.  把从**客户端**收到的所有数据，原封不动 `write` 给**目标服务器**。
2.  把从**目标服务器**收到的所有数据，原封不动 `write` 回给**客户端**。

## 总结：VER 字段的多重身份

在编写代码时，通过 `VER` 字段可以判断当前所处的阶段：

*   收到 `0x05`：可能是**握手第一步**，也可能是**认证后的请求指令**。
*   收到 `0x01`：处于**用户名密码校验**的子协议中。
*   后续数据：如果连接已建立，数据可能是加密的 HTTPS 流量或纯文本。
